Part I

Product

What we're building, who it's for, and the end-to-end journeys customers will experience. Read this part if you care about what the platform does and how it feels to use.

Context

LLMs increasingly generate HTML reports as their final artifact: data analyses, dashboards, write-ups. Today the workflow is:

  1. LLM produces an HTML file locally.
  2. User opens it in a browser to read it.
  3. User shares it — emails the file, generates a PDF, or pastes into a Google Doc.
  4. Report changes — user must re-distribute the new version and chase down everyone who has the old one.

The problem is distribution and updates, not generation. Existing competitors (Linkyhost, HTMLSave, generic file hosts) all miss the LLM-native workflow: no programmatic upload from agents, no stable URLs that auto-update when the report is re-uploaded, no team/org model, no folder organization, no flexible sharing controls.

Goal. A SaaS platform where an LLM (or any client) can push an HTML report to a user's account via API/MCP, the user gets a permanent shareable URL, and re-uploads update the live version at that URL without breaking links. Built B2B-ready (orgs from day 1), starts free, designed for a Stripe-based paid tier later, security-first from line one.

User-facing summary

  • Login & orgs. Sign in via Clerk. Every user has a personal org by default and can create/join more.
  • Upload. From the web (drag-and-drop) or programmatically (HTTP API with an API key, MCP server, or Claude Code Skill).
  • Stable URLs. Each report has a permanent slug. Re-uploading creates a new version but the public URL doesn't change.
  • Folders. Organize reports in a folder tree inside the org.
  • Sharing modes (per report). public link · password-protected link · org-members-only · email allowlist (magic link).
  • Collaboration (per folder). Grant write access to anyone by email — including users outside your org. Grants inherit. Collaborators upload via their own agent.

Security commitments user-visible

These are the public-facing promises we make to users and that should appear in the marketing site, signup flow, and docs:

  • MFA required for org admins. Two-factor auth is enforced through Clerk for any user with admin privileges.
  • Every upload is scanned before it goes live. Malware (ClamAV) + phishing heuristics + cryptominer detection. Flagged uploads show a 451 interstitial; the URL never serves harmful content.
  • Default CSP blocks uploaded JS from calling external servers. If a report tries to phone home to an attacker's site, the browser blocks it. Power users on paid plans can opt-in per report.
  • One-click API key revocation with last_used_at, geo, and rate visible in your dashboard. We alert you on anomalies (new country, rate spikes, repeated failures).
  • Per-page "Report abuse" link with a documented takedown SLA. Anonymous reporting; IP-hashed; appeals workflow.

Architecture overview — hexagonal pattern

We adopt hexagonal architecture (Cockburn's "ports and adapters"): a pure domain core surrounded by ports (interfaces) and adapters (concrete implementations). The domain knows nothing about Remix, Drizzle, Clerk, or R2 — only about Reports, Versions, Folders, and the rules governing them. Every infrastructure choice is replaceable behind a port.

Layer responsibilities

Domain core (pure)

Plain TypeScript. Zero infrastructure imports. Entities, value objects, domain services, domain events. Pure functions; deterministic; trivially unit-testable. Lives in packages/domain/.

Use Cases (interactors)

One class per intent (UploadReportUseCase, SetAclUseCase, …). Takes a DTO, orchestrates domain logic + outbound ports, returns a DTO. Single public method (execute()). Lives in packages/application/.

Ports (interfaces)

TypeScript interfaces the domain depends on. Repositories, gateways, event bus, blob storage. No concrete code. Lives next to the domain code that uses them.

Adapters (concrete)

Drizzle implementations of repositories, R2 client wrapper, Clerk gateway, Upstash limiter, ClamAV queue client, Resend notifier. Each adapter is a thin translation layer. Lives in packages/adapters/.

What this buys us

  • Testability. Use cases tested with in-memory fakes for every port. Domain code tested with zero mocks.
  • Replaceable infrastructure. Swap Neon for Supabase, R2 for S3, Clerk for WorkOS — touch only the adapter, never the domain.
  • Security audit clarity. All side-effecting code lives in adapters; all rules live in the domain. Reviewing the rules doesn't require reading SQL.
  • Forced separation of concerns. Hexagonal stops "let me just add this Drizzle call inside the React component" before it gets started.

Transactional outbox pattern

We use the Postgres transactional outbox to publish events atomically with state changes. No extra queue dependency for the critical path; at-least-once delivery; ordered per aggregate.

Design rules for events

  • Events are facts, named in past tense: ReportVersionUploaded, not UploadReportVersion.
  • Payloads carry just enough for handlers to act — usually aggregate id and the new state. Handlers re-fetch if they need more.
  • Idempotent handlers. Every handler tolerates being invoked twice for the same event (we get at-least-once).
  • One commit, one event. Don't emit two events from one transaction if they should be ordered — emit one and let handlers fan out.
  • No event-driven cycles. A handler can produce a new event but not one already on the call chain. We compile-time-check this with a static analyzer in CI.

Design patterns

DTO (Data Transfer Object)

Every boundary between adapters and use cases passes plain TypeScript objects validated by Zod schemas. DTOs are inputs and outputs; never domain entities. Naming: UploadReportInput, UploadReportOutput, ReportSummaryDto. The same Zod schema is used to validate the HTTP body, the MCP tool input, and the unit test fixture — one source of truth.

Use Case (interactor)

One class per intent, single public execute(input: InputDto): Promise<OutputDto> method. Receives outbound ports via constructor (DI). Contains no framework imports, no HTTP details, no SQL. Easy to test with fake ports.

// packages/application/src/upload-report.use-case.ts
export class UploadReportUseCase {
  constructor(
    private readonly reports: ReportRepository,
    private readonly versions: VersionRepository,
    private readonly blobs: BlobStorage,
    private readonly events: EventBus,
    private readonly permissions: PermissionResolver,
    private readonly limits: PlanLimiter,
  ) {}

  async execute(input: UploadReportInput): Promise<UploadReportOutput> {
    await this.permissions.assertCanWrite(input.actorUserId, input.folderId);
    await this.limits.assertWithinPlan(input.orgId, 'reports.write');
    // ... extract zip, store blobs, create version row, emit event ...
    return { slug, viewUrl, version, scanStatus: 'pending' };
  }
}

Domain Service

Domain logic that doesn't belong to a single entity. PermissionResolver walks the folder ancestry and checks folder_collaborators. ScanPipeline coordinates the sequence of scanners. AnomalyDetector evaluates an ApiKeyUsed event against historical baselines. Pure functions where possible.

Repository

Interface lives in the domain (ReportRepository). Drizzle implementation lives in the adapter layer (DrizzleReportRepository). Repositories return entities, not Drizzle rows. The translation happens inside the adapter via a mapper function.

Gateway (anti-corruption layer)

External services are wrapped: ClerkIdentityGateway, R2BlobStorage, UpstashRateLimiter, ResendNotifier. The wrapper translates external types into our types. We never leak Clerk's OrganizationMembership shape into a use case.

Result type (functional error handling)

Use cases return Result<T, AppError> for expected failure (e.g. NotAllowed, PlanLimitExceeded, ScanFlagged) instead of throwing. Unexpected failures still throw and become 500s. The HTTP adapter maps AppError kinds to status codes in one place.

Specification (for permission rules)

canWrite(user, folder) is a Specification that composes simpler checks: IsOrgMember(folder.orgId) OR HasFolderGrantOnAncestor(folder, user). New rules compose without rewriting callers.

Engineering practices how we work

Six concrete commitments that govern how code arrives in this repository. Each ships in Phase 0 with skills, hooks, and CI gates so the practice is enforced, not aspirational. The practices are designed for a team that mixes humans and LLM-driven agents — everything below applies equally to either.

1. Strict Test-Driven Development /tdd skill

All implementation follows Red → Green → Refactor. Pattern mirrors the proven setup at zora-pantheon: a user-invocable skill (/tdd <task>) walks the cycle, and a Stop hook blocks completion if any affected package has failing tests. There is no path to "done" with red tests.

Workflow

  • Bug fix path (Prove-It). Write a failing test that reproduces the bug → fix → harden with edge-case tests. The reproduction test is the first commit.
  • New feature path. Phase 0 Understand → Phase 1 Red (write failing tests covering happy + edge + error) → Phase 2 Green (minimal implementation, no test edits) → Phase 3 Refactor (improve, keep tests green).

3. Functional programming (vanilla TypeScript) no new libs

Domain and application layers are written in a functional style using only TypeScript and the standard library — no fp-ts, Effect, Remeda, Ramda, or Immer. The discipline is small enough to keep in the team's head and avoids tying the codebase to a heavy abstraction tax.

The four rules

Pure functions in the domain

Domain functions take inputs, return outputs, no side effects, no globals, no Date.now(), no Math.random(). Clock and randomness injected via outbound ports.

Immutability

readonly on every type. ReadonlyArray instead of T[]. as const on literals. Never mutate inputs — always return a new value. CI lints for .push/.splice/assignment on domain types.

Composition over inheritance

No abstract classes in the domain. Functions compose via pipe() (a 12-line utility, no library). Discriminated unions instead of class hierarchies. Use cases compose pure functions with effectful ports.

Side effects at the edge

All I/O lives in adapters. The domain returns "what should happen" (a DomainResult with events to publish); the use case interprets that and calls ports. Easy to test; easy to audit.

What this looks like in code

// packages/domain/src/services/permission-resolver.ts
type Folder = { readonly id: string; readonly orgId: string; readonly parentId: string | null };
type Grant  = { readonly folderId: string; readonly granteeUserId: string;
                readonly permission: 'editor' | 'admin' };

export const findAncestorGrant = (
  folder: Folder,
  ancestors: ReadonlyArray<Folder>,
  grants:    ReadonlyArray<Grant>,
  userId:    string,
): Grant | null =>
  [folder, ...ancestors]
    .flatMap(f => grants.filter(g => g.folderId === f.id))
    .find(g => g.granteeUserId === userId) ?? null;

// Pure. No mocks needed to test. No new libraries.
  • Classes are still used for use cases (they hold injected dependencies) and adapters (they wrap stateful infrastructure). The class is a thin shell; the logic inside is pipelines of pure functions.
  • Result<T, E> type is hand-rolled (~15 lines), used uniformly across use cases.
  • Branding for value-object types (type Slug = string & { readonly _: 'Slug' }) prevents accidental string mixing without a runtime cost.

4. Git worktree workflow for agent-driven work parallel safety

Any time an agent (or human) starts a new feature, bug fix, refactor, or experiment, the work happens in a dedicated git worktree, not by switching branches in the main checkout. This lets multiple in-flight changes coexist without stash drama, lets agents work in parallel without stepping on each other, and keeps the main working tree clean for review and ad-hoc debugging.

Convention

# Start a feature (from the project root)
git worktree add worktree/<feature-slug> -b feat/<feature-slug>
cd worktree/<feature-slug>
pnpm install
# ... do work, commit, push, open PR ...

# When merged, clean up
git worktree remove worktree/<feature-slug>
git branch -D feat/<feature-slug>

Encoded in CLAUDE.md (agent instructions)

  • Before starting any feature, bug fix, or experiment, the agent runs git worktree add worktree/<slug> -b <type>/<slug> from the project root. Never edits files in the root checkout for in-progress work.
  • One worktree per task. Don't reuse a worktree for unrelated work.
  • Worktrees live under worktree/ inside the project (gitignored) — keeps everything for one project grouped under one tree, and ls ~/Projects stays clean.
  • A /worktree-cleanup skill lists stale worktrees (merged or abandoned for >14 days) and removes them after confirmation.

Source control & branching

  • Repository: github.com/agranado2k/<repo> (public).
  • Default branch: main. Protected. No direct pushes. Linear history.
  • Branch naming (enforced by CI regex check):
    • feat/<slug> — new behavior
    • fix/<slug> — bug fix
    • refactor/<slug> — no behavior change
    • chore/<slug> — deps, tooling, CI config
    • docs/<slug> — docs-only changes
  • Commit messages: Conventional Commits encouraged (feat:, fix:, chore:, docs:); commitlint configured but not blocking in v1.
  • Worktree-first: every branch starts as a worktree (ADR-025). The main checkout is read-only for in-progress work.

Branch protection on main

Configured under Settings → Branches → Branch protection rules. The intent is "nothing reaches main without a PR, a human review by the repo owner, and a green pipeline."

  • Require a pull request before merging.
  • Require approvals: 1, from CODEOWNERS (the user). PR authors cannot approve their own PR.
  • Dismiss stale approvals when new commits are pushed (re-review every change).
  • Require status checks to pass before merging (see CI table below). Branches must be up to date with main before merging.
  • Require linear history (squash-only merges — one commit per PR on main).
  • Restrict who can push to matching branches: nobody (force everything through PRs, including the owner).
  • Disallow force pushes and disallow deletions.
  • Include administrators in the rules (so the owner can't accidentally bypass).
  • Require signed commits on main (GPG/SSH-signed).

Cross-cutting schema policies

  • UUIDv7 for ids (time-ordered, index-friendly). Slugs are separate from ids and are URL-facing.
  • Soft delete (deleted_at TIMESTAMPTZ NULL) on all user-visible resources. Hard purge runs as a scheduled job after the appeal window.
  • Tenant isolation enforced in repositories: every query filters by org_id. A linter (custom ESLint rule) flags raw db.select().from(reports) without an org_id filter.
  • JSONB usage only for shapes we never query individually (manifest_json, scan_findings, meta_json, plan_limits_json). Anything we want to filter/aggregate gets a real column.
  • Foreign-key cascades: ON DELETE RESTRICT by default; ON DELETE CASCADE only on report_versions → reports and acls → reports. The app does soft-deletes; cascades are a defense for accidental hard-deletes via migrations.
  • Partitioning: audit_log and outbox partitioned monthly. Old partitions detached and cold-exported.
  • Migrations: Drizzle migrations; every PR includes the up + down. CI runs migrations against a temporary Neon branch and rolls back.

Upload flow (HTTP API — source of truth)

POST /api/v1/reports
  Auth: Bearer <api_key>
  Body (multipart/form-data):
    file: <single .html OR .zip bundle>
    title?: string
    folder_path?: string
    update_slug?: string
    acl?: { mode, password?, allowed_emails? }
    csp_extras?: { connect_src?: [...] }   # paid-plan only
  Response 201:
    { slug, view_url, version, scan_status, dashboard_url }
  1. Authenticate API key → resolve acting_user_id + scopes. requireScope("reports:write").
  2. Resolve target folder.
  3. canWrite(user, folder): org member OR ancestor has folder grant → allowed; else 403.
  4. If update_slug: load report, verify folder check covers it.
  5. Sync pre-checks: size, file count, MIME content-sniff, zip-slip, decompression-ratio, SVG rejection.
  6. Extract + upload to R2 at reports/<reportId>/v<n>/<path>.
  7. Insert report_versions with scan_status='pending'; insert outbox row for ReportVersionUploaded. Single transaction.
  8. Do not bump live_version_id — scanner promotes on clean.
  9. Upsert acls. Return slug + viewer URL + scan_status.

Viewer flow (view.<domain>)

  1. Request view.<domain>/<slug> (optionally ?v=N, ?token=…).
  2. Vercel Edge Middleware: per-IP rate limit; reject unknown slug; return 403 on Service-Worker: script; scan-status pre-fetch — if flagged, render 451 without loader.
  3. Remix loader on routes/$slug.$.tsx: ACL gate (public / password / org / allowlist).
  4. Resolve live_version_id (or ?v=N); fetch from R2; stream via Response with correct content-type.
  5. Apply full security-header stack (ADR-013). Inject abuse-report link.

Viewer at view.<domain>, app at app.<domain>. Viewer never receives app auth cookies. ADR-002.

Service workers blocked at edge. Short Cache-Control. SVG rejected. Trusted Types on dashboard. CSP report-only shadow policy.

ADR-017 Terraform for all long-lived infrastructure Accepted new infra

Context The platform spans several long-lived dependencies (Vercel, Neon, Cloudflare/R2, Upstash, Clerk, Resend, GitHub repo settings). Click-ops causes drift, makes recovery hard, makes per-env reproducibility impossible, and creates single-person knowledge dependencies. Phase 0 explicitly delivers reproducible infrastructure before feature code so the rest of v1 builds on a stable, audited foundation.

Decision

  • Every long-lived resource is provisioned via Terraform under infra/terraform/.
  • Per-provider modules: vercel-app, neon-project, r2, cloudflare-zone, upstash-redis, clerk-app, github-repo (branch protection, secrets, CODEOWNERS), resend-domain.
  • Envs (prod / staging / shared) compose modules. shared/ holds single-instance resources.
  • After bootstrap, all infra changes go through a PR: CI runs terraform plan; CD runs terraform apply on merge — identical workflow to application code, identical review by CODEOWNERS + Claude + Gemini.
  • Click-ops remains only for (1) per-provider PAT generation and (2) the bootstrap R2 bucket for Terraform state (chicken-and-egg).
  • Ephemeral per-PR Neon branches use the Neon API directly from CI — Terraform is too slow for ephemeral state. Prefix isolation handles R2 (pr-<N>/) and Upstash (pr-<N>:).

Alternatives

  • Pulumi (TypeScript IaC) → declined; user requested Terraform; ecosystem maturity favors Terraform for Cloudflare/Vercel/Neon providers.
  • Click-ops + runbook → declined; survives no team handoff or incident drill.
  • Crossplane / k8s-native IaC → declined as drastically over-scoped for our footprint.

Consequences One-time bootstrap pain (~1-2 days). After bootstrap, every change is reviewed, planned, and applied via PR. Disaster recovery from "I deleted the wrong thing" is a terraform apply. Onboarding a second engineer is clone + read docs/infra.md + tf.sh plan. Infrastructure code is itself versioned, blamed, and AI-reviewed.

ADR-018 Terraform state on Cloudflare R2 with Postgres advisory locking Accepted new infra

Context ADR-004 already commits us to R2 for blob storage; concentrating in Cloudflare reduces account sprawl. Terraform's s3 backend supports R2 natively (S3-compatible API). The remaining question is locking: R2 lacks DynamoDB-equivalent conditional writes, so the standard dynamodb_table lock isn't available. Without locking, parallel terraform apply calls (e.g. from two agents in parallel worktrees) corrupt state.

Decision

  • State files in a dedicated R2 bucket tf-state (versioning on, separate keys per env: prod.tfstate, staging.tfstate, shared.tfstate).
  • Server-side encryption at rest is R2's default; access via R2 API token scoped to this bucket only.
  • A wrapper script infra/terraform/scripts/tf.sh acquires a Postgres advisory lock on Neon (pg_advisory_lock(hashtext('tf-' || $env))) before invoking terraform plan|apply, releases after.
  • We already pay for Neon; the lock adds zero new dependencies.
  • Bootstrap order: (1) operator creates the tf-state R2 bucket via Cloudflare dashboard; (2) operator runs tf.sh init configuring the backend; (3) all subsequent infra (including application R2 buckets) is Terraform-applied.

Alternatives

  • HCP Terraform Cloud → rejected; adds a third-party hosted dependency; conflicts with the "stay in Cloudflare" preference.
  • No locking, solo-trust → rejected; parallel agent worktrees will collide.
  • Cloudflare D1 lock table → considered; equivalent quality; chose Postgres because we already use it and the lock semantics are battle-tested.
  • Local state + git-crypt → rejected; not concurrency-safe; loses history.

Consequences A ~30-line shell wrapper to maintain. Recovery from a stuck lock is one psql query (SELECT pg_advisory_unlock(...)). State history via R2 versioning makes "I rolled back the wrong thing" recoverable. All TF secrets stay in GitHub Actions Secrets and a gitignored .tfvars.local for the bootstrap operator.

ADR-019 Infrastructure-first delivery — skeleton + E2E harness before feature code Accepted new delivery

Context Starting feature code before infrastructure and the test harness exist creates the perennial "is it my code or is it the environment?" debugging cycle — it eats more time than infra-first work costs. We want to invert: every environment real, every CI/CD pipeline exercised, and a green E2E harness validating foundations before line one of feature code.

Decision

  • Phase 0 splits into 5 sub-phases (0a-0e): accounts + TF state bootstrap (0a), Terraform modules + apply (0b), skeleton apps + CI/CD (0c), E2E harness with 13 infrastructure tests passing green (0d), engineering practices wired (0e).
  • Skeleton apps include /health routes that exercise Neon + R2 + Upstash so infrastructure tests have something concrete to assert against. Full secureHeaders() stack applied. Edge MW skeletons in place (rate-limit stub, Service-Worker: → 403 stub).
  • Use-case Gherkin features for Phase 1+ are written in Phase 0 and tagged @pending. Playwright skips them until their use case is implemented. The contract is the spec; BDD enforces it.
  • By end of Phase 0: 13 infrastructure tests green; CI/CD has run end-to-end against real infrastructure; feature code starts in Phase 1 with confidence that any failing test is the code, not the environment.

Alternatives

  • Start coding features in Phase 1, provision infra on demand → rejected; "is it my code or env" debugging eats more time than infra-first costs.
  • Mock all infrastructure in tests → rejected; integration bugs only surface against real services; we'd need this harness for v1.1 anyway.
  • Skip pre-feature E2E entirely → rejected; the team explicitly asked for "e2e before code."

Consequences Phase 0 grows from ~1 week to ~1.5-2 weeks of foundational work. After that, every Phase 1+ PR runs in a real environment with fast feedback. The infrastructure tests double as a prod smoke suite (cron-scheduled post-launch).

ADR-020 Hexagonal architecture (ports & adapters) Accepted new arch

Context The platform spans multiple infrastructure dependencies (Postgres, R2, Clerk, Upstash, ClamAV, Resend). Without an architectural boundary, business rules and SQL/framework code will tangle quickly, making testing slow and security review hard.

Decision

  • Domain core (packages/domain): entities, value objects, domain services, domain events. Zero infra imports.
  • Application layer (packages/application): use cases (interactors). One class per intent. Receives outbound ports via constructor DI.
  • Adapters (packages/adapters): concrete implementations of ports — Drizzle repositories, R2 storage, Clerk gateway, Upstash limiter, ClamAV queue, Resend notifier.
  • Driving adapters (in the Remix apps): HTTP routes, MCP server, admin UI — each calls use cases through their input DTO.

Alternatives considered

  • Classic MVC with services everywhere → rejected; the domain rules get pulled into the framework layer over time.
  • Full DDD with aggregates as classes + event sourcing → rejected as overkill for v1.
  • No deliberate architecture → rejected; security and test discipline degrade quickly.

Consequences

  • Use cases are testable with in-memory fakes; no DB needed for application-layer tests.
  • Swapping any infrastructure (Neon→Supabase, Clerk→WorkOS) touches one adapter.
  • Slight upfront ceremony cost — mitigated by code generation for repository boilerplate.
  • Security review is faster: rules are concentrated in the domain; SQL doesn't hide invariants.

ADR-021 Event-driven via Postgres transactional outbox Accepted new arch

Context Several flows are inherently asynchronous (scan, notify, takedown grace period, audit). We need atomic emission of events with state changes, plus at-least-once delivery, without adding a new infrastructure dependency in v1.

Decision

  • Use cases write event rows to a Postgres outbox table inside the same transaction as the state change.
  • A dispatcher worker (Vercel Cron tick + QStash for scheduled/delayed) polls outbox every ~1s, dispatches to in-process handlers, marks delivered.
  • All handlers must be idempotent (we get at-least-once).
  • Events are facts in past tense; minimal payloads; no event-driven cycles (CI-enforced).

Alternatives considered

  • External queue (SQS, RabbitMQ) → rejected for v1; extra dependency, no transactional guarantee.
  • Direct synchronous calls everywhere → rejected; ties unrelated contexts together and slows responses.
  • Postgres LISTEN/NOTIFY → rejected as primary mechanism; doesn't survive worker restarts, no replay.
  • Full event sourcing → rejected as scope creep.

Consequences

  • Zero new infra in v1; the outbox is a single Postgres table.
  • Monthly partitioning of the outbox keeps the hot set small.
  • Scaling beyond a single worker is straightforward (claim rows with FOR UPDATE SKIP LOCKED).
  • If we ever need stricter ordering, we already have it — outbox rows are inserted in transaction order per aggregate.

ADR-022 Strict TDD with hook-enforced gates Accepted new practice

Context A platform that hosts third-party HTML and gates security at every layer cannot survive an "I'll add tests later" culture. Test discipline must be structural, not aspirational, and equally applicable to LLM-driven contributions.

Decision Strict Red-Green-Refactor TDD, enforced by Claude Code hooks mirrored from zora-pantheon:

  • /tdd <task> skill walks the cycle (with a separate "Prove-It" path for bug fixes).
  • Stop hook (tdd-stop-guard.sh) blocks turn completion if any affected package has failing tests.
  • PreToolUse hook (tdd-protect-tests.sh) warns when editing test files — nudges toward fixing implementation.
  • PostToolUse hook (tdd-post-edit-tests.sh) runs affected-package tests after every source edit.
  • Rules files (tdd-test-files.md, tdd-source-files.md) attached to test/source paths.
  • CI greps for .skip(), .only(), TODO in test files; fails on any.

Alternatives "Tests-after" (rejected, no enforcement). External CI-only gate (rejected, too late in the loop). Property-based testing as primary (deferred to v1.1 for domain functions).

Consequences Bigger upfront discipline cost; pays back in confidence at refactor time. Hooks add ~5s to each agent turn but prevent regressions. Domain + application layers reach 100% line/branch coverage naturally; adapters covered pragmatically.

ADR-023 BDD with Gherkin for use-case e2e tests Accepted new practice

Context Use cases are the unit of value — the surface where product intent meets technical implementation. We need executable specifications in business language that survive refactors and are reviewable by non-engineers.

Decision One .feature file per use case in tests/e2e/features/, executed with Playwright + playwright-bdd. Features cover happy paths, the main alternative flows, and security-relevant negative paths. PRs that add or change a use case must also add or update its feature. CI fails if a use case lacks a feature.

Alternatives

  • Cypress + Cucumber → rejected; Playwright is stronger for our cross-origin tests (viewer vs app).
  • No e2e, rely on unit + manual → rejected; can't catch ACL / origin-isolation regressions.

Consequences Slower CI (each feature spins a real browser). Mitigated by parallelization (Playwright shards) and Neon-branch databases. Worth it for the regression catch on cross-origin / ACL / CSP behavior — the things hardest to spot in unit tests.

ADR-024 Functional programming style, no new libraries Accepted new practice

Context Hexagonal architecture wants pure domain logic. We want immutability, composability, and testability without paying the cognitive cost of adopting fp-ts, Effect, or similar full-fat FP libraries — those are powerful but raise the bar for every new contributor (human or agent).

Decision Pure functions in domain + application layers; readonly everywhere; composition over inheritance; side effects pushed to adapters. Zero new dependencies — only TypeScript and the standard library. A 12-line pipe() and a 15-line Result<T, E> live in packages/domain/src/fp/. Branded types for value objects (zero runtime cost).

Alternatives

  • fp-ts / Effect → rejected for v1; large surface area, opinionated, raises the agent learning curve.
  • Remeda / Ramda → rejected; adds a dependency for what stdlib mostly already does well in TypeScript.
  • Pure OOP with rich domain objects → rejected; harder to test, encourages hidden state.

Consequences Onboarding stays low-friction. CI lints flag mutation on domain types (no-param-reassign, custom rule against .push on readonly arrays). If we ever need richer FP machinery, the shape of our code makes migrating to Effect straightforward (it's still pipelines).

ADR-025 Git worktrees for all in-progress agent work Accepted new practice

Context Agents do multiple things in parallel. Branch-switching in a single checkout causes node_modules churn, build-cache invalidation, and tooling confusion. It also makes it easy for an agent to accidentally cross-contaminate two in-flight changes.

Decision Every new feature, bug fix, refactor, or experiment is performed in a dedicated worktree under ../<feature-slug>, on a branch named <type>/<slug>. The convention is codified in CLAUDE.md and a /worktree-cleanup skill prunes stale worktrees. Main checkout is reserved for review and ad-hoc inspection.

Alternatives Single-checkout + branch switching (rejected; causes cache churn and cross-contamination). Per-task containers (rejected as v1 overkill).

Consequences Disk usage grows with active tasks (a worktree shares .git but not node_modules). Mitigated by pnpm's content-addressed store. Massive isolation win when an agent is running multi-step work in the background while another task is on the main checkout.

ADR-026 Documentation-as-contract pre-push hook Accepted new practice

Context The single most common failure mode in fast-moving SaaS projects is doc rot: schema changes ship without updating db-design.md; ADRs stop being added; READMEs go stale. Once docs lag, every future contributor (human or agent) is working from incorrect assumptions.

Decision A pre-push gate (both a git hook and a Claude Code PreToolUse hook for git push commands) inspects the diff against a trigger matrix (schema → db-design.md; new use case → feature file + README; new ADR file → INDEX.md link; etc.) and blocks the push if the matched docs aren't part of the change. A /docs-check skill walks the agent through the matrix.

Alternatives Trust the team / agent (rejected; doesn't survive month two). Mandatory PR checklist (rejected; easy to ignore). Linter-as-warning-only (rejected; warnings are noise).

Consequences Friction on first encounter, then it becomes muscle memory. Emergency bypass exists (PUSH_WITHOUT_DOCS=1) but is logged and flagged in the PR — cheap to use, visible cost. Docs stay accurate by construction.

ADR-027 OpenAPI 3.1 source of truth + Bruno collections Accepted new practice

Context Per ADR-003, the HTTP API is the source of truth and the MCP server is a thin client. The contract must be machine-readable (for the MCP server's runtime validation), human-readable (for our docs site), agent-friendly (for LLMs reading it), and git-versioned (no cloud-only artifacts).

Decision

  • OpenAPI 3.1 at docs/api/openapi.yaml is the canonical spec. Routes are validated against it at boot in dev/CI.
  • Bruno for desktop client collections. Generated from OpenAPI via gen-bruno-from-openapi.ts; committed under docs/api/bruno/; environments per stage.
  • Scalar mounted at app.<domain>/docs renders the OpenAPI for human browsers.

Alternatives

  • Postman → rejected; cloud lock-in, weak git story, paid for team features.
  • Insomnia → rejected; recent versions require Kong cloud account.
  • Hoppscotch → viable alternative; chose Bruno for stronger plain-text-file format and PR diffability.
  • OpenAPI-only without a desktop client → rejected; engineers want a click-and-send tool for exploration.

ADR-028 GitHub Flow with protected main, PR-only delivery Accepted new delivery

Context Solo (for now) build with heavy LLM-assisted contributions. Without structural enforcement, "I'll just push that small fix" becomes the norm; main becomes unreviewable; rollbacks get harder. We want a workflow that protects the integrity of main even from the owner.

Decision

  • Public repository at github.com/agranado2k/<repo>.
  • main is protected: no direct pushes, no force pushes, no deletions, linear history (squash-only), signed commits required.
  • Every change goes through a PR from a typed feature branch (feat/· fix/· refactor/· chore/· docs/).
  • Required: 1 approval from CODEOWNERS; stale approvals dismissed on new commits; up-to-date with main before merging.
  • "Include administrators" is on — the owner cannot bypass the rules. Emergency bypass only via explicit "Disable protection" with audit trail.

Alternatives

  • Trunk-based with direct push for the owner → rejected; defeats the audit trail and AI-review loop.
  • GitFlow (release branches) → rejected as overkill for a SaaS that continuously deploys.
  • Allow self-approval for the owner → rejected; the AI reviews + CI gate are the floor, but at least one human review keeps the spec aligned.

Consequences Marginal friction for trivial changes; large friction for big ones (which is correct). The protected-main rule applies to everyone, which means even hotfixes go through a PR — mitigated by fast CI and the local pre-push parity.

ADR-029 GitHub Actions for CI on PR + CD on merge Accepted new delivery

Context We already use GitHub for source control. Putting CI/CD on GitHub Actions keeps the toolchain coherent (one place for secrets, one place to read logs, one set of permissions to reason about), avoids a vendor like CircleCI/Buildkite, and gets us preview deploys for free via Vercel's GitHub integration.

Decision

  • CI (ci.yml) on pull_request: biome, typecheck, branch-name, unit tests, migration check (Neon branch), e2e (Playwright + BDD against per-PR Vercel preview), security-headers curl, Bruno contract tests, docs-trigger-matrix. All are required status checks on main.
  • CD (cd.yml) on push to main: deploy app + view to Vercel prod (commit-pinned), push scanner image to GHCR, terraform apply, run migrations, smoke tests, notify.
  • Preview deploys are produced by Vercel's GitHub integration on every PR; they use the ephemeral Neon branch + the CI R2 bucket.
  • Concurrency: CI cancels in-progress runs per PR (latest push wins); CD never cancels (we want every commit on main to deploy).

Alternatives

  • CircleCI / Buildkite → rejected; extra surface, no advantage for our workload.
  • Vercel-only pipelines → rejected; insufficient for non-Vercel jobs (Terraform, GHCR, e2e orchestration).
  • Deploy on tag instead of merge → rejected; we want continuous deploy, not staged release.

Consequences Strong single-vendor coupling (acceptable; if we ever leave GitHub, Actions YAML translates to most competitors). Cost is low at v1 scale. Required-checks list is long — mitigated by parallel job graph and aggressive concurrency.

ADR-030 Dual AI PR review (Claude + Gemini) as advisory checks Accepted new delivery

Context A solo + LLM-assisted project has no second pair of human eyes by default. AI review provides a credible "second reviewer" voice on every PR without blocking velocity, catches a different class of issue than CI (architectural drift, security reasoning, idiomatic suggestions), and importantly produces written feedback that becomes a learning artifact for both the owner and future agents.

Decision

  • Two independent reviewers via GitHub Actions: Claude (anthropics/claude-code-action@v1, model claude-opus-4-7) and Gemini (google-github-actions/gemini-cli@v1, model gemini-2.5-pro).
  • Both post inline comments and a summary. Reviews are advisory — they do not satisfy the CODEOWNERS approval requirement.
  • Specialized focus: Claude is tasked with security headers, ACL logic, CSP changes, schema-migration safety, TDD compliance, FP discipline, ADR alignment. Gemini is tasked with code quality, idiomatic TypeScript, performance, readability.
  • Either reviewer can post "Request changes"; that adds a visible "Changes requested" status until the author addresses or dismisses (does not block merge once CODEOWNERS approves).
  • Cost guardrail: skip on diffs >4,000 lines (split-the-PR signal).

Alternatives

  • Single AI reviewer → rejected; two models disagreeing is the most useful signal.
  • Make AI approval blocking → rejected; AI false positives shouldn't gate merge.
  • GitHub App-based reviewers (no Actions) → viable; we chose Actions for unified secret management and per-PR cost visibility.

Consequences Marginal API spend per PR. Two bots posting comments creates noise on small PRs — mitigated by tuning the focus prompts and skipping on trivial-diff heuristics. The owner still owns the merge decision; AI reviews are a force multiplier, not a replacement.
