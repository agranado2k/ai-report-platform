# ADR-0063: In-viewer editing on the viewer origin

- **Status**: Accepted ŌĆö implementation gated on a `/security-review` pass before the implementation PR ships (amends ADR-0038)
- **Date**: 2026-07-07
- **Deciders**: agranado2k
- **Relates to / amends**: ADR-0038 (report viewer access & serving ŌĆö **amends** it by adding a second, authenticated route profile to the viewer origin), ADR-002/ADR-013 (viewer origin isolation + security-header stack, unchanged for the public route), ADR-014 (service-worker block, unchanged), ADR-0037/ADR-0062 (upload/versioning pipeline + the ProseMirror editing model this route serves), ADR-0056 (report sharing & ACLs ŌĆö access-token/unlock-cookie precedent this ADR's edit token rhymes with), ADR-0059/ADR-0060 (owner + write-grant authorization, re-checked per save), ADR-0051 (MCP OAuth 2.1 / API-key auth ŌĆö no anonymous writes on any surface), ADR-012/ADR-0045 (scan gate ŌĆö every save re-enters it).

## Context and problem statement

`view.<domain>/<slug>` (ADR-002/0038) is a deliberately credential-free, PSL-isolated origin that serves raw, untrusted report bytes under a strict sandbox CSP with no first-party JS and the service-worker block (ADR-014). That isolation is the whole security model: a malicious report can't reach app credentials or persist itself, because there's nothing on that origin to reach.

The editing epic (ADR-0062: a ProseMirror-based editor for report content, informed by the `spike/DECISION.md` verdict) wants users to edit their report *in place*, ideally without leaving the viewer's rendered context. That means putting an authenticated, first-party JS application ŌĆö the editor ŌĆö on the same origin that today exists specifically to have none. This is the central tension this ADR resolves, and it was flagged explicitly in PR #144 review: **does the enforcing/sandbox CSP have to be relaxed for the editor, and if so, how is that contained?**

## Decision drivers

- Preserve the ADR-002/013/0038 isolation guarantee for the existing public read path ŌĆö zero regression, zero exception.
- Give the editor a first-party JS surface without turning the whole viewer origin into an authenticated app.
- Never let the editor execute the untrusted report bytes it's editing ŌĆö those bytes are exactly what the isolation model doesn't trust.
- Reuse proven machinery (ADR-0056's app-mints/viewer-verifies token pattern) instead of inventing a new trust primitive.
- Every write path (dashboard, viewer edit route, MCP, ADR-0051) enforces auth and `canWrite` (ADR-0059/0060) server-side ŌĆö UI is never the boundary.
- Ship gated on independent security review, given the sensitivity of adding auth + JS to an intentionally-bare origin.

## Considered options

- **Two CSP profiles on one origin, edit route only** *(chosen)* ŌĆö public route (`GET /<slug>`) stays exactly as today; a new `GET /<slug>/edit` route serves the editor under a separate, controlled CSP. Only the edit route ever gains first-party JS.
- **Relax the public route's CSP to allow the editor bundle everywhere** (rejected ŌĆö reopens the sandbox for every reader of every report, not just an authenticated owner/grantee actively editing; the isolation guarantee would degrade to "usually not exploited" instead of "structurally can't be").
- **Move editing entirely to the dashboard origin (`app.<domain>`), no in-viewer editing** (documented as the fallback ŌĆö see Decision 3 ŌĆö preserved as the answer if security review rejects the in-viewer route).
- **Full Clerk session on the viewer origin** (rejected ŌĆö see Decision 3).

## Decision outcome

### 1. The CSP question, answered up front

The enforcing/sandbox CSP for the **public** route does **not** relax, at all. A second, separate CSP profile is introduced, scoped **only** to the new `GET /<slug>/edit` route:

- `script-src 'self'` ŌĆö first-party editor JS only, nothing report-supplied; no `unsafe-inline`, no `unsafe-eval`. (Superseded the original "plus build-time hashes" wording: the implementation ships the *stricter* bare `'self'`, since this repo's Remix/React stack already hydrates without inline scripts ŌĆö see `appHeaders()`. A per-response nonce, not hashes, is the documented path if a future editor build step ever needs an inline allowance; flagged for `/security-review`.)
- `connect-src` limited to the app-origin API (`app.<domain>/api/v1/...` ŌĆö the token-mint and save endpoints), nothing else.
- `worker-src`, `frame-ancestors`, `object-src`, and the rest of the ADR-013 stack carry over from the public profile ŌĆö the edit route is *additive* (it gains a script allowance the public route doesn't have), never a general loosening.
- The two profiles are selected by route, computed independently, and there is no code path where a public `GET /<slug>` request can receive the edit-route CSP or vice versa. e2e security-header tests assert both profiles independently (see Consequences).

This containment is the answer to the PR #144 question: yes, the sandbox CSP is relaxed, but *only* for one authenticated route, *only* for a first-party bundle, and the public read path ŌĆö still the overwhelming majority of viewer traffic ŌĆö is untouched byte-for-byte.

### 2. Two route profiles, never mixed

- **`GET /<slug>` (public, unchanged)**: no auth, no first-party JS, the existing strict sandbox CSP, the ADR-014 service-worker block, the full ADR-0038 state machine (200/404/410/451, `noindex`). Nothing in this ADR touches this route's behavior or headers.
- **`GET /<slug>/edit` (new, authenticated)**: serves the editor application bundle under the CSP profile in Decision 1. Requires a valid edit token (Decision 3). The edit route is the **only** place the viewer origin gains first-party JS or an authenticated session concept ŌĆö no other route on `view.<domain>` changes.

### 3. Auth: a scoped edit token, minted by the app

Chosen mechanism, deliberately rhyming with ADR-0056's access-token / unlock-cookie pattern: the dashboard (`app.<domain>`, Clerk session) mints a **short-lived, single-report, single-purpose edit token** when a user with `canWrite` on a report (ADR-0059/0060: `isOwner OR hasWriteGrant`) opens the editor. The viewer's edit route validates the token's signature and scope (slug-bound, purpose-bound, exp-bounded ŌĆö same HMAC-compact codec family as the `Access token`), then **re-checks `canWrite` server-side on every save**, not just at token-mint time ŌĆö a grant revoked mid-session stops working on the next write, not just the next token mint.

Considered and rejected:
- **Full Clerk session on the viewer origin** ŌĆö would put the entire authenticated session (cookies, whatever Clerk needs client-side) on the untrusted-content origin, enlarging the blast radius of any future viewer-origin compromise to "attacker can act as the signed-in user," not just "attacker can read/write via a narrow, purpose-bound token." Rejected outright, independent of the security-review outcome.
- **Dashboard-origin editing with a deep-link fallback** ŌĆö edit entirely on `app.<domain>` (which already has Clerk + full first-party JS), with the viewer's `/edit` route (if it exists at all) merely deep-linking there. Documented here as **the fallback if `/security-review` rejects in-viewer editing** ŌĆö it preserves the UX goal (one click from the report to editing it) at the cost of leaving the viewer origin's read-only posture completely alone, at zero new viewer auth surface.

### 4. Saves re-enter the existing pipeline ŌĆö no mutate-in-place

An edit-save does not mutate stored bytes directly. It produces a **new `ReportVersion`** through the same R2-first/commit-last pipeline as any other upload (ADR-0037, and the versioning extension in ADR-0062): write to R2, run the ADR-012 scan gate, and only promote to `live_version_id` on a `clean` verdict (ADR-0045's async pipeline). A malicious or corrupted edit cannot go live unscanned ŌĆö the scan gate that already protects uploads protects edits identically, with no special-case bypass for the edit route.

### 5. No anonymous writes, on any surface

Enforcement of auth and `canWrite` lives server-side in the use cases (401 unauthenticated / 403 not-authorized), independent of transport: the REST API, the new viewer edit route, and MCP (ADR-0051 ŌĆö every MCP write call carries OAuth 2.1 or an `arp_` API-key Bearer) all funnel through the same authorization check. UI-level hiding of the edit affordance is never treated as a security boundary. Correspondingly, the **public** `GET /<slug>` route never serves editor JS, comment data (ADR-0064), or any edit affordance ŌĆö those exist only behind `/edit`.

### 6. ADR-014 is unchanged, including for future collaboration

The service-worker registration block stays in force on the viewer origin, for both routes, with no carve-out for the edit route. This includes any future real-time collaboration transport (ADR-0067, not yet written) ŌĆö a collab layer must not register a service worker on `view.<domain>` to get around this; if it needs background delivery, it does so through the app-origin API, not a viewer-origin service worker.

## Consequences

**Positive**

- The public read path ŌĆö still most of the traffic ŌĆö is provably untouched: same CSP, same headers, same state machine, same absence of JS.
- The isolation break is contained to exactly one route, one CSP profile, and one narrowly-scoped token ŌĆö not a general credential.
- Saves reuse the existing scan-gated, versioned pipeline; there is no new "fast path" that skips moderation.
- Rhyming the edit token with the ADR-0056 access-token codec means no new signing primitive, just a new claims shape and purpose tag.

**Negative / risk delta (stated honestly)**

- The viewer origin gains an authenticated, first-party-JS-bearing route for the first time since ADR-002. This is a real, non-zero increase in that origin's attack surface, even though it's contained to `/edit`. That is exactly why this ADR's Status is gated on an independent `/security-review` pass before the implementation PR ships ŌĆö this document records the intended design, not a green light to build without that review.
- e2e security-header tests must now cover **both** CSP profiles explicitly (today's suite only asserts the public stack, ADR-013) ŌĆö a regression that silently applies the edit-route CSP to the public route (or vice versa) must fail CI.
- If a report is ever submitted to the Public Suffix List (PSL) as a distinct isolation boundary, the existence of an authenticated sub-route under the same eTLD+1 as the public content is a fact that submission reviewers may reasonably ask about; this ADR is the reference for how the exception is bounded.
- The dashboard-origin fallback (Decision 3) is a real, ready-to-ship alternative if security review finds the in-viewer token model unacceptable ŌĆö it is not a hypothetical, it should be treated as a live option during that review.

## More information

- Implementation: the edit-route CSP profile and its e2e header assertions live alongside the existing ADR-013 `secureHeaders()` helper (as a second, route-selected profile, not a fork); the edit-token codec extends the ADR-0056 `packages/domain/src/signed-token.ts` shared primitive with an `edit` claims shape (slug, purpose, exp) parallel to `AccessClaims`.
- **Edit-token codec ŌĆö implemented (Phase 2, `packages/domain/src/edit-token.ts`)**: `EditClaims { slug, exp, sub, scope: "edit" }`, built on the shared `signed-token.ts` primitive, mirroring `access-token.ts`'s structure exactly (`mintEditToken`/`readEditToken`/`verifyEditToken`). Single-report via `slug`; short-lived via `exp` (suggested 15ŌĆō30 min TTL ŌĆö shorter than the owner `Access token`'s 24h, since edit is a higher-privilege capability than read; the exact constant is deferred to the mint site); carries the acting user via `sub` (required, non-empty ŌĆö an edit token with no bound subject isn't a valid capability); single-purpose via the `scope: "edit"` literal discriminant, which is the security boundary that prevents token confusion ŌĆö `parseEditClaims` is strict enough that a validly-signed `Access token` (even an `owner: true` one, sharing the same HMAC secret) cannot narrow into `EditClaims`, because `AccessClaims` has neither `sub` nor `scope`. Verified by a dedicated cross-parse-rejection test (`edit-token.test.ts`) that mints a real access token via `mintAccessToken` and asserts `readEditToken` rejects it. The view origin verifies signature + scope only (never holds Clerk creds); `canWrite` re-check on save and the `/edit` route wiring are still pending (next phase), along with the app-side mint-into-the-open-flow.
- **Edit-route CSP profile ŌĆö implemented (Phase 3, `packages/headers/src/view-headers.ts`)**: `editViewHeaders({ appOrigin, reportToUrl? }): Headers`, a second, route-selected profile alongside the existing `viewHeaders()` (the public profile's byte-for-byte output is unchanged ŌĆö pinned by a regression test). `appOrigin` is REQUIRED (the app-origin API the edit token authenticates against). Both profiles share their unrelated directives (`default-src`, `img-src`, `font-src`, `frame-ancestors`, `base-uri`, `form-action`, `object-src`, `worker-src`, `report-to`) via one internal `CSP_SHARED` table, so they can't silently drift on the directives that are supposed to stay identical. Every relaxation vs the public profile, and its rationale:
  - **No top-level `sandbox` CSP header** ŌĆö the single biggest relaxation. The public profile pairs its enforcing CSP with a second `sandbox` header because the top-level document IS the untrusted report; the edit route's top-level document is instead the TRUSTED first-party editor app (Remix/React), which needs same-origin DOM/storage and to call the app-origin API, so sandboxing it would break it and buys nothing security-wise the isolation model needs. Containment of the untrusted report being edited does NOT depend on this: the report renders inside the editor's own sandboxed `srcDoc` iframe (`sandbox="allow-same-origin"`, no `allow-scripts`), which carries its own, independently enforced, more restrictive `<meta>` CSP (`default-src 'none'; style-src 'unsafe-inline'; img-src data: ŌĆ”` ŌĆö `apps/app/app/editor/iframe-document.ts`).
  - **`connect-src 'self' <appOrigin>`** ŌĆö widened by exactly one explicit, caller-supplied origin (never `'*'`), so the editor's first-party JS can reach the edit-token-authenticated app-origin API.
  - **`script-src 'self'`** ŌĆö NOT loosened to `'unsafe-inline'`. This mirrors `appHeaders()` (`packages/headers/src/app-headers.ts`), which already ships `script-src 'self'` with no `'unsafe-inline'` for the same Remix/React stack on the dashboard origin, i.e. this repo's own Remix build doesn't need an inline-script allowance to hydrate. No `'unsafe-inline'` was needed.
  - **`frame-src 'self'`** ŌĆö a NEW directive the public profile doesn't have at all, required so the editor's own `srcDoc` report iframe is permitted to render (browsers resolve a `srcDoc` frame's effective origin to its embedder's for `frame-src` matching purposes; no `blob:`/`data:` needed since the iframe is only ever given `srcDoc` markup, never a `blob:`/`data:` URL).
  - **`Cache-Control: no-store`** (vs the public route's `private, max-age=60, must-revalidate`) ŌĆö authenticated, per-user route; nothing about it should be cached.
  - Everything else ŌĆö `default-src 'self'`, `style-src 'self' 'unsafe-inline'` (Tailwind), `img-src`, `font-src`, `frame-ancestors 'none'`, `base-uri 'none'`, `form-action 'self'`, `object-src 'none'`, `worker-src 'self'`, the report-only shadow policy, COOP/CORP/Origin-Agent-Cluster/Referrer-Policy/Permissions-Policy/nosniff/HSTS/Report-To ŌĆö is IDENTICAL to the public profile: additive, never a general loosening, matching this ADR's Decision 1.
  - **OPEN for `/security-review`** (flagged, not resolved by this phase): (1) whether "top-level document not sandboxed" is acceptable given the editor is the first `first-party-JS-bearing`, authenticated route ever added to `view.<domain>` (the Consequences section's stated risk delta); (2) confirm in real target browsers that `frame-src 'self'` is sufficient for the `srcDoc` iframe (historically inconsistent across implementations) before this profile is wired to a route. `script-src 'unsafe-inline'` was NOT used, so it is not an open item ŌĆö noted here only because the implementation prompt asked to flag it if used.
  - **No route wires this profile yet.** `editViewHeaders` is a pure, unit-tested header builder with no caller ŌĆö it is inert in production until the `/edit` route (Phase 4) selects it, and the full `/security-review` pass this ADR's Status gates on runs against that wired surface, not this function in isolation.
- **Signing-secret trade-off (flagged, not decided)**: should the edit token use a signing secret separate from the `Access token`'s (`VIEW_ACCESS_TOKEN_SECRET`), for defense-in-depth key separation between the read/share and edit capabilities? Default taken here is to **reuse the shared secret**, because the `scope: "edit"` discriminant already makes cross-token confusion structurally impossible at the parse layer (see above) ŌĆö a separate secret would add operational surface (another rotation, another env var, another place the app and view must agree) without closing a gap the scope check doesn't already close. The case *for* a separate secret: if the shared secret ever leaks, blast radius is identical either way (an attacker who can forge one can forge the other), so key separation only helps if the two secrets are rotated/scoped independently in practice (e.g. edit token issuance restricted to a narrower app code path). Left for the reviewer/mint-phase implementer to weigh; no runtime behavior depends on this yet since minting doesn't ship in this phase.
- ADR-0062 owns the ProseMirror schema and the parse-to-model / model-to-HTML boundary this route relies on (Decision 2 of this ADR: the editor renders the model, never the raw blob).
- **Edit-token API acceptance seam ŌĆö implemented (this API slice, HTTP endpoints for Decision 3's dashboard fallback)**: the `canWrite` re-check on save (Decision 3/4) and "no anonymous writes, on any surface" (Decision 5) are now wired end-to-end on the app-origin REST API, so a cross-origin caller on `view.<domain>` (the dashboard-fallback deep link's eventual in-viewer editor client, or any bearer of a minted edit token) can drive an edit session entirely over HTTP:
  - **Acceptance boundary**: `resolveEditTokenActor` (`apps/app/app/server/edit-token-actor.server.ts`) is the LAST front door tried by both `resolveUploadActor` and `resolveActorForRead` (`apps/app/app/server/auth.server.ts`), gated on the route actually having a `:slug` param ŌĆö a route with no slug (e.g. `/api/v1/keys`) never reaches it, so it can't become a general Clerk bypass. It re-verifies the token's signature/scope/expiry AND re-runs `loadWritableReport`'s LIVE `canWrite` check (a grant revoked after mint rejects on the very next call, not just the next mint) ŌĆö never trusts the token's mere validity as proof of CURRENT authorization.
  - **Scope, deliberately narrow**: an edit-token actor is granted `["reports:write"]` ONLY ŌĆö never `acl:write`. It can save content, list/read versions, and read/write comments on the report it's bound to; it can NOT grant/revoke write access, read/write the ACL, move, rename, or delete the report, or act on any other report (the token is slug-bound; a route's raw `:slug` path segment must be the CANONICAL slug the token was minted for, or `resolveEditTokenActor` rejects the mismatch).
  - **Endpoints accepting the edit token** (all under `/api/v1/reports/{slug}/...`, `{slug}` = the canonical slug): `POST .../versions` (save an edit as a new `ReportVersion`, `origin: 'editor'` ŌĆö never a create path, always via `updateSlug`), `GET .../diff?from=<version_id>&to=<version_id>` (JSON structural/fallback diff between two versions ŌĆö **note for the 4c client**: the diff payload contains HTML *derived from stored report bodies*, so when the view-origin editor renders it, it MUST flow through the same sandboxed-iframe + CSP the viewer applies (ADR-013) and the service-worker block (ADR-014); rendering it unsandboxed or on the app origin would reintroduce exactly the XSS/cookie-theft exposure the view-origin split exists to prevent ŌĆö claude-review #183 F-1), `GET .../comments` + `POST .../comments` (list / add / reply), `PATCH .../comments/{comment_id}` (resolve), `GET .../versions` (list). The comments/versions GET+POST surfaces were already auth-agnostic (built on the `resolveActorForRead`/`resolveUploadActor` seam before this slice existed) ŌĆö this slice added CORS to them and confirmed the edit-token path end-to-end; the save and diff operations are new.
  - **CORS posture (security-critical)**: `Access-Control-Allow-Origin` is the EXACT configured `VIEW_ORIGIN` (read from `defineEnv()`, never the request-origin-fallback `viewOrigin(request)` helper used for building absolute URLs elsewhere ŌĆö that fallback would make the allow-list trivially self-satisfying), echoed only when the request `Origin` matches byte-for-byte ŌĆö never `*`, never a reflected arbitrary origin, and never any origin at all when `VIEW_ORIGIN` is unset (fails closed, same posture as the edit-token secret itself being unset). `Access-Control-Allow-Credentials` is NEVER set. `Access-Control-Allow-Headers: Authorization, Content-Type` and a resource-specific `Access-Control-Allow-Methods` answer the `OPTIONS` preflight (204, no body) BEFORE any auth runs ŌĆö a preflight carries no `Authorization` header by construction, so it must never touch the DB. The preflight advertises `Access-Control-Max-Age: 600` (caches only the origin-independent method/header allow-list ŌĆö never a credential ŌĆö so the editor skips a redundant round-trip per call). The wrapper **merges** `Vary: Origin` into any `Vary` the handler already set rather than overwriting it (a clobber could drop a handler's own `Vary: Accept-Encoding` and poison a shared cache). Implemented as a pure header-decision module (`packages/http/src/cors.ts`) plus a route wrapper (`apps/app/app/server/cors.server.ts`, `corsRoute`), unit-tested for the matching/non-matching/preflight/`Vary`-merge matrix.
  - **Why no CSRF concern despite the relaxed CORS**: auth here rides a custom `Authorization: Bearer <editToken>` header, never an ambient cookie. A cross-site request forger can make the browser SEND a request, but cannot make it carry an app-origin bearer token it has no way to read ŌĆö and `Access-Control-Allow-Credentials` being unset means even a same-origin-looking cookie-based session never rides along on a cross-origin call to these routes. This is the same rationale ADR-0056's access-token / unlock-cookie precedent established for token-based capabilities.

  - **New: `POST /api/v1/reports/{slug}/edit-token` ŌĆö the silent-refresh backend.** An editing session's `scope:"edit"` token is short-lived (15 min) by design (ADR-0063 Decision 3's TTL rationale: a write capability should live a fraction as long as a read one). Implementation: `refreshEditToken` (`apps/app/app/server/edit-token-refresh.server.ts`) re-runs `loadWritableReport` a SECOND time against the actor `resolveEditTokenActor` already re-verified upstream ŌĆö belt-and-braces, mirroring EXACTLY how `reassembleAndSaveEditedVersion` (the save path) re-checks canWrite even though its callers already resolved a canWrite-derived actor. `sub` on the newly-minted token is always the FRESHLY re-checked actor's `userId` ŌĆö there is no old-token claims input to this endpoint at all (the route only ever receives the already-authenticated actor + slug from the `handle()`/`resolveUploadActor` seam, never the raw presented token).     - **SECURITY TRADE-OFF (operator-approved "sessions never break," flagged for `/security-review`):** making an edit token refreshable for as long as `canWrite` holds means a LEAKED token is ALSO refreshable ŌĆö its effective life extends indefinitely, bounded only by how long the thief keeps calling this endpoint before the grant is revoked or ownership changes, not by the original 15-min TTL. Mitigations, all already in place and independent of this endpoint: **(a)** single-report, `reports:write`-only scope (`EDIT_TOKEN_SCOPES`, `auth.server.ts`) ŌĆö never `acl:write`, so a thief can edit that one report's content, nothing else (can't reshare, delete, move, or touch any other report); **(b)** the LIVE `canWrite` re-check on EVERY refresh (not just the original mint) ŌĆö revoking the grant or transferring ownership cuts the thief off within one remaining TTL (Ōēż15 min) of their last successful refresh, not up to 24h; **(c)** the token only ever leaves the app on the view origin's strict edit-route CSP (Decision 1) + sandboxed report iframe (┬¦F-1) ŌĆö the control that's actually supposed to keep it out of an attacker's reach in the first place, this endpoint doesn't add a new place for it to leak from.
    - **Recommendation, NOT built (left for the operator/security-review to weigh):** an ABSOLUTE session cap ŌĆö "never valid more than N hours after the FIRST mint of this editing session, no matter how many times it's refreshed" ŌĆö independent of and additional to mitigation (b) above (which bounds exposure PER-REVOCATION-LATENCY, not total session length; a grant that's never revoked could in principle be refreshed forever). Implementing it cleanly would need a `sessionStartedAt` (or similar) claim threaded through every mint/refresh, so refresh #2 still remembers refresh #1's origin, plus a cap check in `refreshEditToken` comparing it to `nowSeconds()` ŌĆö a real, if small, change to `EditClaims` (`packages/domain/src/edit-token.ts`), which this task was explicitly scoped NOT to touch. Not built here; flagged for the reviewer to decide whether (b) alone is sufficient.
