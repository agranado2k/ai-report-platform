# ADR-0052: Stripe-style prefixed external IDs + third-party ID segregation

- **Status**: Accepted
- **Date**: 2026-06-22
- **Deciders**: agranado2k
- **Supersedes / amends**: —
- **Relates to**: ADR-0036 (DDD value objects; `UserId`/`OrgId` shared kernel), ADR-001 / ADR-0038 (`Slug` as capability), ADR-0048 (Clerk provisioning), ADR-0051 (MCP tools that carry these ids).

## Context and problem statement

Two related gaps in how entity identifiers cross the API / MCP boundary surfaced while planning sharing/ACL (which will expose more ids):

1. **Raw, inconsistent ids on the wire.** Every PK is **UUIDv7** (generated by `UuidV7IdGenerator`), so there is *no* sequential/enumerable-id risk. But the exposed forms are inconsistent and not self-describing: **folders** expose a **raw UUID** (`id`, `parent_id`, `folder_id`), while **reports** are addressed by a **`nanoid(10)` slug** (a capability, not a discoverable id — ADR-001/0038). The exposed UUID *is* the internal PK; there is no external/internal boundary. A consumer holding `019ed70f-…` cannot tell it's a folder, and nothing stops it being passed where a report id is expected.

2. **Third-party (Clerk) ids conflated with ours at the type/glossary layer.** The schema already segregates them — `users.id` (our uuid PK) vs `users.clerk_user_id` (Clerk's id, separate `text` column, unique); `orgs` likewise; all FKs reference *our* uuid. **But** the Clerk ids are typed as plain `string` (`ClerkIdentity.clerkUserId: string`) while our `UserId`/`OrgId` are branded, and the glossary misleadingly says *"UserId … originates in Clerk; mirrored on `users.id`"*. The conflation invites accidentally exposing or persisting a Clerk id as if it were ours.

We want the convention fixed **before** sharing/ACL adds user/org/grant ids to the wire.

## Decision drivers

- Self-describing, mistype-proof identifiers on the wire (Stripe ergonomics: `cus_…`, `re_…`).
- A clean **external/internal** boundary and a clean **third-party/own** boundary — never leak an internal PK shape or a vendor id.
- Minimal migration; keep UUIDv7 PKs and their index performance.
- Set one convention now so sharing/ACL inherits it.

## Considered options

- **Encoding after the prefix**: (a) **base62 of the existing UUIDv7** — reversible codec, no migration *(chosen)*; (b) a separate opaque random `external_id` column — Stripe-true, no time-order leak, but migration + backfill + extra lookup; (c) prefix the raw hyphenated uuid as-is — trivial but long/ugly, not Stripe-like.
- **Prefix style**: (a) **long readable** (`report_`, `folder_`, …) — self-evident *(chosen)*; (b) short Stripe codes (`rpt_`, `fld_`, …) — compact but needs a legend.
- **Cutover**: (a) **clean break, prefixed-only** — bare uuid → 422 *(chosen, we own all clients)*; (b) accept-both-in / prefixed-out transition window — safer for third-party consumers, but a dual path to carry and later remove.
- **Reports identity**: (a) **keep `slug` + add a `report_` id** — capability vs addressing roles both kept *(chosen)*; (b) slug-only (no report id) — fewer handles but breaks the "every entity has a prefixed id" convention; (c) replace slug with `report_` — rejected, breaks the ADR-0038 capability-URL model.
- **Third-party ids**: (a) **own branded `ClerkUserId`/`ClerkOrgId` types, never exposed** *(chosen)*; (b) leave them as plain `string` (status quo) — invites conflation with our `UserId`/`OrgId`.

## Decision outcome

1. **Prefixed external ids.** Every exposed entity id is **`<prefix>_<base62(uuidv7)>`** — a **reversible codec over the existing UUIDv7 PK** (no new column, no migration, deterministic round-trip). Prefixes are **long + readable**:

   | Prefix | Entity | Source column |
   |---|---|---|
   | `report_` | Report (alongside its slug) | `reports.id` |
   | `folder_` | Folder (`id`, `parent_id`, `folder_id`) | `folders.id` |
   | `org_` | Org (when exposed) | `orgs.id` |
   | `user_` | User (when exposed — sharing) | `users.id` |
   | `version_` | ReportVersion | `report_versions.id` |
   | `key_` | ApiKey | `api_keys.id` |
   | `grant_` | `Collaborator` — a folder access grant | `folder_collaborators.id` |

   The `grant_` row applies to the **existing `folder_collaborators` table** (glossary term **`Collaborator`**) — it does NOT introduce a new `acl_grants` table. Whether sharing/ACL keeps that table/term or renames it is the sharing/ACL ADR's call (ADR-0036 "one name per concept"); this ADR only reserves the `grant_` prefix for whatever that row ends up being.

2. **The codec lives in the branded value objects.** `make<Entity>Id(s)` *decodes* `prefix_base62` → the internal uuid (validates the prefix; a wrong prefix or a bare uuid → `ValidationError`); a `toWire(id)` *encodes* uuid → the prefixed string. The domain stays uuid internally — the codec is purely the parse/serialize boundary. The branded types stop cross-entity mixups at **compile** time; the prefix stops them at **runtime**.

3. **Clean break — prefixed-only (no transition window).** Input *and* output are prefixed-only; a bare uuid (or a wrong-entity prefix) is a `422 ValidationError`. We own all clients (the MCP server + our scripts), so a one-shot cutover is acceptable and avoids a dual-path forever. (If a 404-style privacy posture for wrong-entity ids is ever wanted, that's a separate change.)

4. **Reports keep the slug *and* gain a `report_` id.** The **slug** stays the **capability / view-URL** handle (ADR-0038 — an unguessable `nanoid(10)` possession-token, *not* a discoverable id; `view.<domain>/<slug>` unchanged). **`report_…`** is the **authenticated API addressing id**. `GET|PATCH|DELETE /api/v1/reports/{handle}` and the MCP report tools accept **either** a `report_` id **or** a slug (detect by prefix); report responses return **both** `id` (`report_…`) and `slug`.

5. **Third-party ids are segregated and never exposed.** Our internal uuid PKs are the **sole** source of our `user_`/`org_`/… ids. Clerk ids stay in the `clerk_user_id`/`clerk_org_id` columns, gain their **own branded types `ClerkUserId` / `ClerkOrgId`** (distinct from `UserId`/`OrgId`), and are **never serialized** on the API/MCP wire. Any future third-party identifier follows the same `<provider><Entity>Id` field-and-type pattern (e.g. a `StripeCustomerId`).

## Consequences

- **Good:** self-describing wire ids; a folder id can't be passed where a report id is expected (prefix + brand); clean internal/external *and* vendor/own boundaries; **zero migration** (codec over the existing uuid); the convention is ready before sharing/ACL.
- **Trade-offs:**
  - `base62(uuidv7)` still embeds UUIDv7's **creation-time ordering** (a low-sensitivity leak — same as today's raw uuid; not enumeration). If that ever matters, swap to a separate random `external_id` column — the codec boundary keeps that change localized.
  - **Clean break** means a one-shot update of the MCP tool inputs (`folder_id`/`parent_id`) + any scripts. Accepted (we own them).
  - **Two handles per report** (`id` + `slug`) — small cognitive cost, justified by the capability-vs-addressing distinction.
- **Surface that changes (implementation PR):** openapi `format: uuid` → the prefixed-string pattern + examples (Bruno regen); MCP tool schemas/descriptions; the http mappers (encode on output) and route parsers (decode on input, dual-resolve report handle); new `ClerkUserId`/`ClerkOrgId` brands threaded through `ClerkIdentity` + the provisioner ports.

## Implementation notes (a follow-up PR — NOT this ADR)

- **Fixed-width base62 (round-trip safety).** Naive base62 of the 128-bit integer drops leading-zero bytes → variable-length, ambiguous decode. The codec MUST left-pad to a canonical fixed width (22 chars for 128 bits) so `make*Id(toWire(id)) === id` holds — including a uuid with leading zero bytes (`00000000-…`). Assert this round-trip in a test.
- **Report handle detection is structural, not `startsWith("report_")`.** nanoid's alphabet includes `_`/`-`, so a 10-char slug *could* begin with `report_`. Discriminate by shape: a valid external id is `report_` + 22 base62 chars (len 29) vs a slug of exactly 10 — not the prefix alone.
- **ADR-024 applies** (the codec lands in `packages/domain`): vanilla TS + the existing `Result<T,E>`, no new FP libs, `readonly` throughout, decode failures as `ValidationError` (not exceptions).
- `packages/domain`: the base62⇄uuid codec + prefix table; `make*Id`/`toWire` per entity; `ClerkUserId`/`ClerkOrgId` brands + constructors (these names must appear in code to satisfy the glossary's "introduce a term where it's first used" rule).
- `packages/http` + `apps/app/app/routes/api.v1.*`: encode outputs, decode inputs, dual-resolve the report handle.
- `apps/mcp`: tool input/output (prefixed folder ids; `reports_*` accept `report_` id or slug).
- `docs/api/openapi.yaml`: replace `{ type: string, format: uuid }` id fields with the prefixed pattern + examples.
- `docs/domain-glossary.md`: correct `UserId`/`OrgId`; add `External Id`, `ClerkUserId`/`ClerkOrgId`, and the prefix table.
