# Domain glossary — Ubiquitous Language

The registry of canonical terms for Centaur Spec, per **ADR-0036** (Domain-Driven Design). Use these spellings and meanings consistently across code (type names, function names), commit messages, PR titles, ADRs, the diary, and conversations.

**Product name** — the product is **Centaur** (the short wordmark used in the app chrome — `TopBar`/`Logo` and page-title suffixes) and **Centaur Spec** (the full name used in standalone copy, docs, and the viewer/MCP/OpenAPI surfaces). Domain: `centaurspec.com`. This is a brand name, **not** a bounded-context domain term and not used in code: the workspace packages (`arp-*`), the GitHub repo, and the Neon project keep the original `ai-report-platform` technical identifier (PR #126).

**Adding a term**: introduce it in the same PR that first uses it in code. Cross-reference the spec section that defines its behavior. Prefer business names over technical ones (`slug`, not `id`).

**Changing a term**: rename across the codebase in a single PR. Update this glossary in the same commit. Do NOT leave aliases — the point of Ubiquitous Language is that there is exactly one name per concept.

> **Source of truth.** This glossary, `docs/context-map.md`, and ADR-0036 are the canonical source for domain *language*. Where `docs/spec.html` (rev 7) disagrees, this glossary wins and the spec is synced to it (see the diary entry "Domain-language alignment — spec rev 8 sync"). The spec still wins on *architecture* decisions; this carve-out is for naming only.

## Reports & Folders context

- **Report** — a versioned HTML document published to the platform under a permanent slug. Lives in an `Org` (tenancy), is owned by its **Owner** (the creating `User`, ADR-0059), and is located under a `Folder`. Aggregate root. Spec: ADR-001 + Architecture / Data model.
- **Owner (of a Report)** — the `User` who created the report (`reports.owner_id` — lands with migration 0010, ADR-0059; the `uploadReport` actor, backfilled from the version-1 uploader). The only principal who may delete, `set_acl`, or manage `Write grant`s; rename/re-upload/move additionally allow write-grantees (`canWrite = isOwner OR hasWriteGrant`, ADR-0059/0060). Org members see org reports' metadata in lists but get no content or write access from membership alone — org admins included. Amends ADR-0056's "owner = any member of the owning org". Spec: ADR-0059.
- **Slug** — the permanent, URL-safe `nanoid(10)` identifier for a `Report`. Globally unique across the platform. Cannot change after creation. Value Object. In `public` ACL mode the slug is also the access **capability** (~10¹⁸ entropy = unguessable "anyone with the link"); it is not a discoverable id. Spec: ADR-001, ADR-0038.
- **ReportVersion** — a single snapshot of a `Report`'s content. A `Report` has one or more `ReportVersion`s and points to a `live_version_id`. Re-uploading creates a new `ReportVersion`, never a new `Report`. Part of the `Report` aggregate. Persisted in `report_versions`. Spec: ADR-001.
- **ReportSummary** — a lightweight read projection of a `Report` for list views (the dashboard + `GET /api/v1/reports`): `slug`, `title`, `isPublished` (a clean version is live), and `folderId` (its `Folder` placement; `folder_id` on the wire). NOT the full aggregate — listing must not load every `ReportVersion`'s manifest. An application read type (like a DTO), not an aggregate.
  - _Avoid_: Version (ambiguous on its own — always `ReportVersion`).
- **ReportSearchQuery** — the parameters of an org-wide report **search**: an optional case-insensitive `query` (matched as a literal substring against title + slug), an optional `folderId` filter, and `Cursor pagination` params (`limit` + `startingAfter`/`endingBefore`, ADR-0053). An application read type. Spec: ADR-0036, ADR-0053.
- **ReportPage** — a `CursorPage` of `ReportSummary`s: `items` + `hasMore` (ADR-0053 cursor pagination — no `total`). The result type of the `searchReports` use case. An application read type. Spec: ADR-0036, ADR-0053.
- **ReportVersionSummary** — a lightweight read projection of a `ReportVersion` for the version-history list (`GET /api/v1/reports/{slug}/versions`): `versionNo`, `versionId`, `uploadedBy`, `uploadedAt`, `scanStatus`, `sizeBytes`, `origin`. NOT the full `ReportVersion` — listing must not load manifests. An application read type, analogous to `ReportSummary`. Spec: ADR-0065.
- **Search (reports)** — the dashboard's org-wide, newest-**created**-first, cursor-paginated listing of `Report`s with an optional title/slug filter (`searchReports` use case → `ReportRepository.searchByOrg`, keyset on the report `id` — ADR-0053). Org-scoped: a caller only ever sees their own org's reports.
- **Entry document** — the file served at the bare `view.<domain>/<slug>` (the root of a `ReportVersion`'s bundle). Resolved by the `index.html` convention: a single `.html` upload is normalized to `index.html`, and a single wrapping directory in the archive is stripped. Spec: ADR-007, ADR-0037.
- **Live version** — the `ReportVersion` served at the base viewer URL (`view.<domain>/<slug>`). Older versions remain reachable at `?v=N`. Tracked by `Report.live_version_id`.
- **Scan status** — a Value Object on a `ReportVersion` recording the moderation verdict: `pending`, `clean`, `flagged`, `blocked`. Only a `clean` version may become the `live_version_id`. It is a **denormalized cache** of the `ScanJob` outcome (the `ScanJob` aggregate is owned by Abuse & Moderation); Reports & Folders updates it on the `ReportVersionScanned` event. Spec: ADR-012.
- **VersionOrigin** — a Value Object on a `ReportVersion` recording how the version was produced: `upload` (the API/MCP upload path) or `editor` (an in-app Editing session save — not yet built). Carried on `ReportVersionUploaded` for audit/analytics only; no consumer behavior differs by origin. Spec: ADR-0062 §6, ADR-0065.
- **Folder** — a tree-structured organizing unit inside an `Org`. Folders nest. A `Report` belongs to exactly one `Folder`. Aggregate root. Folders stay **org-scoped** (no per-user owner, ADR-0059) — the shared org tree is what team orgs want; content privacy is enforced per-report by the `Acl`. Spec: Architecture / Data model.
- **Root folder** — the single top-level `Folder` of an `Org` (`parent_id = NULL`). A `Report` created without an explicit `folder_path` is placed here. Spec: ADR-0037.
- **Move (a Report)** — reassigning a `Report` to a different `Folder` within the report's `Org` (its `folder_id` is mutable post-create). A `Report` still belongs to exactly one `Folder`. Authorized via `canWrite` (`Owner` or `Write grant`, ADR-0059/0060 — today that means the actor's own org; once write grants land, a cross-org grantee may move it too), and the target folder must be in the **report's** `Org`, not the actor's. Spec: ADR-0036, ADR-0059/0060.
- **Write grant** — the owner-granted, per-report permission that lets another `User` rename, re-upload, or move a specific `Report` (NOT delete / `set_acl` / grant management — those stay `Owner`-only). Persisted in `report_write_grants` (lands with the write-grants build, ADR-0060), keyed `(report, grantee_email)` (normalized `EmailAddress`), with `grantee_user_id` resolved lazily at check time. Works **cross-org** — the grantee is typically outside the report's org. Confers no view access by itself — the read capability (`Acl`) and the write capability are deliberately separate. Supersedes the folder-level `Collaborator`. Spec: ADR-0060.
- **Collaborator** *(superseded by ADR-0060 — see `Write grant`)* — the never-implemented folder-level grant of ADR-009 / ADR-0056 P4 (`folder_collaborators`, inherited down the folder tree, carrying a `Grant level`). The table remains in the schema, unused, until a cleanup migration.
- **Grant level** *(superseded by ADR-0060)* — the `editor`/`admin` Value Object of the folder `Collaborator` design. `Write grant`s have a single implicit level; the `grant_level` enum stays unused.
- **Acl** (Access Control List) — per-`Report` sharing configuration. Part of the `Report` aggregate (one `Acl` per `Report`); loaded on single-report reads, **defaults to `private` when no `acls` row exists** (private-by-default; sharing is an explicit opt-in, ADR-0056). Modes (`AclMode`): `private`, `public`, `password`, `org`, `allowlist`. Spec: ADR-0056 (extends ADR-0038).
- **`private` (mode)** — owner-only sharing (ADR-0056): the viewer serves the report ONLY to an `Access token` carrying the `owner` claim (minted by the app for the report's `Owner`, ADR-0059) — no public link, no password, no allowlist. The **default** for every new report; the owner reaches it one-click via the dashboard's owner-open, everyone else is denied.
  - _Spell out_ "ACL" only when expanding the acronym in prose; the type/term is `Acl`.
- **Access token** — a slug-bound, **mode-bound**, HMAC-signed compact capability **minted by the app** after it authorizes a private-report request (by `Acl` mode) and **verified by the viewer** (ADR-0056). The mechanism that lets the credential-free `view.<domain>` origin (ADR-002/0038) serve a private report without ever holding Clerk credentials. Carries `{slug, exp, mode}` plus, for `allowlist`, the redeemed `email`. `mode` binds it to the Acl mode it authorizes, so a stale cookie can't survive a mode switch. Lifetime is `~15-min` for `password` but equals the grant's `Access expiry` (up to 90d) for `allowlist` — where it is **necessary but not sufficient**: the viewer additionally requires current allowlist membership **and** a live `Report grant` per request (revocation-C). Stateless for `password`/`org`; grant-checked for `allowlist`. An **`owner` claim** (minted by the app at `/reports/{slug}/open` only for the report's `Owner` — the creating user, ADR-0059, previously any member of the owning org) makes the viewer serve regardless of mode — the owner bypasses their own report's share gate (still slug-bound + exp-bounded + signed).
- **Unlock cookie** — a report-scoped (`/<slug>`), short-lived, HttpOnly cookie the **viewer** sets after verifying an `Access token`, so the whole report bundle (entry + relative-URL assets) is gated by one per-request credential (ADR-0056). A self-issued **capability**, NOT app/Clerk credentials — so it doesn't breach the ADR-002/0038 origin isolation.
- **Access expiry** — the owner-configured duration an `allowlist`-mode viewer stays unlocked after redeeming a magic link (`acls.access_ttl_seconds`, ADR-0056). A finite TTL the owner picks (presets within 60s–90d, default 7 days); it sets the `Access token` + `Unlock cookie` lifetime for that grant. Distinct from the fixed ~15-min token `password` mode uses.
- **EmailSender** — the application port for outbound transactional email (`send({to,subject,html})`, ADR-0057). The only adapter is `ResendEmailSender` (Resend HTTP API). Its first use is the `allowlist` magic link. Composition is fail-open (no `RESEND_API_KEY` ⇒ no sender wired, the app still boots), but the unlock route then **fails closed** — the magic link is unavailable rather than silently broken. Distinct from `Notifier` (an unbuilt domain-event sink).
- **NonceStore** — the application port for a single-use, TTL-bounded `id→value` store backing the `allowlist` magic link (ADR-0056). `put` writes with an expiry; `take` is an **atomic get-and-delete** (Redis `GETDEL`) so a nonce redeems exactly once. Adapter: `UpstashNonceStore` (Upstash Redis REST, ADR-0011). Distinct from the stateless, exp-bounded `Access token`.
- **Magic link** — the one-time email link an `allowlist`-mode viewer clicks to prove they own an allowlisted address (ADR-0056). It carries an HMAC-signed **magic-link token** = a `NonceStore` nonce id (the `{slug,email}` payload + ~15-min expiry live in the store, not the URL). Redeeming verifies the HMAC, consumes the nonce (single-use), and mints the durable `Access token` for the owner's `Access expiry`. The link is the *bootstrap*; the grant is the durable record.
- **Report grant** — the durable, revocable record (`report_grants`, the `GrantStore` port) that an `allowlist`-mode viewer holds after redeeming a `Magic link` (ADR-0056, revocation-C). Keyed by `(report, email)`, expiring at `Access expiry`. The viewer checks a **live grant per request**, so the owner removing an email — or switching mode — **deletes** it and revokes access on the next request. Distinct from a `Write grant` (the write capability, ADR-0060) — a `Report grant` is view access only.
- **Signed-token codec** — the shared HMAC-SHA256 compact-token primitive (`packages/domain/src/signed-token.ts`) underlying both the `Access token` and the `Magic link` token (ADR-0056). Owns payload encode/decode, HMAC sign, timing-safe verify, and — for claims-carrying tokens — expiry checking; each token type supplies only its own claims shape/validation (`AccessClaims`'s slug/mode/email/owner fields, or the magic-link's bare nonce id) on top. Extracted from two independently hand-rolled sign/verify implementations so both codecs share one wire format and one constant-time-compare code path (gather-the-sharing-concept refactor).
- **EmailAddress** — a normalized (trim + lowercase) email address Value Object (ADR-0056). The ONE normalization home for `Acl`'s `allowlist` (`allowedEmails`) and the `GrantStore`'s `(report, email)` key — previously implemented independently in three places, including a documented drift bug between two of them (claude-review #114). `normalizeEmailAddress`/`normalizeEmailAddresses` normalize without validating shape (for values already known-shaped, e.g. a stored grant's email); `makeEmailAddress` additionally validates a minimal email shape (non-empty local + domain with a dot) for untrusted input.
- **Report HTML schema** — the constrained ProseMirror node/mark vocabulary a report's `Editable body` is parsed into and serialized from: document structure (`section`, `sec`), blocks (`card`, `checklist`, `resgroup`/`resrow`, `tablewrap`+`table`, `grid`, `details`/`summary`, paragraph-role variants `.desc`/`.lede`/`.sub`), inline marks (`chip`, `pill`, `kbd`, plus standard `strong`/`em`/`a`), and a generic attr-retention rule that preserves any class/attribute not claimed by a named node/mark verbatim rather than dropping it. Spec: ADR-0062 §3.
- **Presentation shell** — the `<head>` + `<style>` half of a report document (fonts, design tokens, the CSS that makes a Centaur report look like one). Opaque to the editor: versioned alongside the `Editable body`, never parsed into the `Report HTML schema`, and re-injected unmodified on export. Spec: ADR-0062 §2.
- **Editable body** — everything inside a report's `<body>`; the only half of the document the editor owns, parsed into and serialized from the `Report HTML schema`. Complements the `Presentation shell`. Spec: ADR-0062 §2.
- **Edit token** — a short-lived, single-report, single-purpose token minted by the app (`app.<domain>`, Clerk session) for a user with `canWrite` on a report — slug-bound (`slug`), scope-bound (`scope: "edit"`, the discriminant that stops an `Access token` from narrowing into an edit capability), exp-bounded (`exp`), carries the acting user (`sub`), same HMAC compact codec family as the `Access token` (ADR-0056). `canWrite` is re-checked server-side LIVE on every accept, not just at mint time. Codec implemented at `packages/domain/src/edit-token.ts` (`EditClaims`/`mintEditToken`/`readEditToken`/`verifyEditToken`) — pure, mirrors `access-token.ts`. Also carries `sessionStart` (epoch seconds, OPTIONAL for backward compat with pre-cap tokens) — the FIRST mint time of a refresh chain, carried forward unchanged by every refresh, which bounds the chain's TOTAL length via the `SESSION_CAP_SECONDS` (8h) **absolute session cap**, independent of write-grant revocation latency (ADR-0063 "Phase 5-D"). **Accepted** (verified + `canWrite`-re-checked) by `resolveEditTokenActor` (`apps/app/app/server/edit-token-actor.server.ts`), the last front door tried by both `resolveUploadActor`/`resolveActorForRead` — this is what lets `app.<domain>`'s own `/api/v1/reports/{slug}/...` REST endpoints (save, diff, comments, versions) accept a `Authorization: Bearer <editToken>` in place of a Clerk session, granting only the `reports:write` scope (never `acl:write`). The viewer-origin `GET /<slug>/edit` route on `view.<domain>` (verifying signature/scope itself, per ADR-0063 Decisions 1-2) is now the sole editing surface — the dashboard-origin editor this API acceptance seam originally fell back to is retired (ADR-0063 "Phase 5-C"); the `/api/v1` endpoints remain the cross-origin data plane the unified experience is built on. Spec: ADR-0063 §3, ADR-0063 "Edit-token API acceptance seam", ADR-0063 "Phase 5-D".

## Identity & Access context

- **User** — an authenticated principal mirrored from Clerk. Identified by a `UserId`. Belongs to one or more `Org`s. Spec: ADR-005.
- **Display name** — a human-readable name for a `User`, mirrored from Clerk at provisioning (`users.display_name`, nullable; `fullName` → `firstName lastName` → `username`, else null; length-bounded at capture). Best-effort and refreshed on re-provision but never cleared except by `User soft-delete`. Distinct from the `User`'s email (also mirrored). Spec: ADR-0048.
- **Author Identity** — a **route-layer projection** (`{ id, email, name }`) attached to a `Comment` / `ReportVersion` on the authenticated, org-scoped API so a reader sees *who* authored an item (preferring `Display name`, then email, then a generic fallback) rather than a raw `user_…` id. Resolved from the mirror per unique author; a soft-deleted or unresolvable author yields nulls. Never surfaced on the public viewer. Not a domain aggregate — it's a presentation DTO over `Identity & Access`. Spec: ADR-0048, ADR-0063.
- **Org** (Organization) — the tenancy unit. Owns folders and API keys, and is the tenancy home of reports (each `Report` additionally has a per-user `Owner`, ADR-0059). A `User` belongs to exactly **one** `Org`, keyed by their email domain (ADR-0068): a public-provider address (gmail, outlook, …) → a **`personal`** org (1:1 with the user, JIT-provisioned per ADR-0048, never gains members); a corporate domain → that domain's **`team`** org (multi-member by design — every same-domain sign-up joins it, JIT). No invitations or org switcher in the first build; multi-org membership is an explicit revisit-later. Kinds persist as `orgs.kind` (ADR-0061 — the column lands with the team-orgs build). Identified by `OrgId`. Carries a `Plan` and its `PlanLimits`. Spec: ADR-005, ADR-0061, ADR-0068.
- **Plan** — the billing tier on an `Org` (enum; e.g. `free`, `pro`). Billing-ready, billing-deferred for v1. Spec: ADR-006.
- **PlanLimits** — a Value Object holding the quota ceilings a `Plan` grants an `Org` (e.g. report count, storage, per-report CSP opt-in). Persisted as `orgs.plan_limits_json`. Enforced by `PlanLimiter` in the Application layer. Spec: ADR-006.
- **ApiKey** — a credential issued to a `User` that acts on their behalf when calling the HTTP API. `arp_live_`/`arp_test_` secret, hashed with **HMAC-SHA-256 + a server pepper** (ADR-0008 amends the spec's argon2id), prefixed for display, scoped per ADR-016. User-scoped (`acting_user_id`) so cross-org collaboration works. Landed for `/api/v1` (alongside Clerk sessions); the MCP server consumes it. The settings UI also labels these **MCP tokens** (since connecting an agent over MCP is the primary use) — a UI synonym for the same `arp_` `ApiKey`, **not** a separate credential type. Spec: ADR-0008.
- **Scope** — a permission carried by an `ApiKey`: `reports:write`, `reports:read`, `folders:write`, `acl:write`. Distinct from a `Collaborator` `Grant level`. Spec: ADR-016.
- **UploadActor** — the resolved caller identity the transport layer presents to the `uploadReport` use case. Carries `UserId`, `OrgId`, `FolderId` (the target folder), and the granted `Scope`s. Resolved by `resolveUploadActor` at the HTTP boundary from the **Clerk session** (ADR-0048): a signed-in session is provisioned (find-or-create the mirrored `User` + personal `Org` + `Root` folder) into an org-scoped actor with `reports:write`; no session → `Unauthenticated` (401). API-key auth (ADR-0008) now resolves the same actor behind the same signature — a Bearer `arp_…` is tried first, else the Clerk session — without the use-case changing. Spec: ADR-016 / ADR-0048.
- **Anomaly** — a read-model over `ApiKey` usage flagged as suspicious by the `AnomalyDetector` (geo shift, rate spike, repeated auth failure). Drives an admin alert; does not block requests synchronously in v1. Spec: ADR-016.
- **AnomalyDetector** — the domain service that evaluates `ApiKeyUsed` events against historical baselines and emits `ApiKeyAnomalyDetected`. Spec: ADR-016.

## Abuse & Moderation context

- **ScanJob** — the aggregate tracking the upload-time content scan of a `ReportVersion` through its lifecycle (`queued` → `running` → `done` / `failed`). Owns the scan `findings` (ClamAV + phishing/miner heuristics). On completion it emits `ReportVersionScanned` carrying the verdict; Reports & Folders caches that verdict on `ReportVersion.scan_status`. Spec: ADR-012.
- **Scanner** — the port (verdict engine) a `ScanJob` runs a `ReportVersion` through to obtain its terminal verdict (`clean` / `flagged` / `blocked`). Phase-1.5a: `CleanStubScanner` always returns `clean` (invite-only MVP); the real ClamAV + phishing/miner engine slots in behind the same port with no call-site change. Spec: ADR-0045.
- **ScanWorkQueue** — the delivery port that hands queued `ReportVersion`s to the async scan worker (the drain) and tracks ack/retry; distinct from `ScanJob`/`scan_jobs`, which remain the source of truth for the cached `scan_status`. Implemented on pg-boss; swappable. Spec: ADR-0045.
- **Abuse report** — a user-submitted complaint about a hosted `Report` (phishing, malware, CSAM, other). Aggregate root. Tracked with `status` and an action audit trail. Spec: ADR-012.
- **Takedown** — the operator action that withdraws a `Report` from public serving. Emits `ReportTakenDown`; Reports & Folders soft-deletes the `Report` row (`deleted_at`) and the R2 keys are queued for purge after the appeal window. Spec: ADR-012.
- **Delete (a Report)** — the **owner-initiated** removal of their own `Report` (vs operator **Takedown**, which is a moderation action). Same `deleted_at` soft-delete mechanism — the viewer then returns 410 and the bytes are retained for the purge window — but a self-service action within Reports & Folders. Unlike Takedown it currently emits no domain event (an intra-context operation with no cross-context consumer yet; cf. moving a report). Spec: ADR-0038.
- **User soft-delete** — mirroring a Clerk `user.deleted` into Identity & Access (ADR-0054): stamp `users.deleted_at`, revoke all the user's API keys, and thereafter resolve the user to **no actor**. **Terminal** — `createPersonalIdentity` refuses to resurrect a soft-deleted user, so a re-auth with the same Clerk identity stays blocked until an explicit (future) restore. The user's `Org` and its org-owned `Report`s are untouched (reports aren't user-owned). Triggered by the Svix-verified `/webhooks/clerk` endpoint; idempotent. Spec: ADR-0054.
- **CSP report** — an inbound Content-Security-Policy violation report sent by viewer browsers to `/csp-report`. Used for policy drift detection. Spec: ADR-013.

## Authoring & Collaboration context

- **Comment** — a workspace collaboration annotation anchored to a location in a report's document. Aggregate root: `id`, `reportId`, `authorUserId`, `body` (bounded length — an annotation, not a document), `anchor` (see `Annotation anchor`), a `resolved`/`open` state, timestamps. Threaded one level — a `Thread` — replies cannot themselves be replied to. Creating, editing, or resolving a `Comment` requires `canWrite` (ADR-0059/0060), not mere read access; the comment's author or the report's `Owner` may resolve/delete it. Never surfaced on the public viewer route — comments are workspace data, not published content. Spec: ADR-0064.
- **Annotation anchor** — the two-part, degrading location a `Comment` attaches to: primarily a relative position into the ProseMirror document (the `Report HTML schema`'s model, ADR-0062) — stable across ordinary edits elsewhere in the doc, and deliberately Yjs-relative-position-compatible so it composes with a future real-time collaboration transport (ADR-0067) without a re-anchoring migration; falling back to a version-pinned snapshot (the `ReportVersion` id + a text-quote snapshot of the anchored content) when the document has changed enough that the relative position can no longer resolve. Degrades to "pinned to version N," surfaced as such in the UI — a comment is never silently dropped or misplaced. Spec: ADR-0064 §2a.
- **Thread** — a root `Comment` plus its replies, single level deep (replies cannot themselves be replied to). Spec: ADR-0064.
- **Comment intent** — a closed value object on every `Comment` signalling what the author wants *done* with it: `note` (the default — a plain human note, no action), `enhancement` (an agent should enhance the anchored context using the comment), `add` (add content per the comment), or `remove` (remove content per the comment). Defaults to `note`: an omitted intent on create/reply resolves to `note`, and a pre-existing comment persisted before the field reads as `note` (backward compat). An explicitly invalid intent on the wire is rejected with a `ValidationError` (422). The agent-action *semantics* that consume a non-`note` intent are deliberately out of scope — the field only carries the signal. Implemented at `packages/domain/src/intent.ts` (`Intent` union + `makeIntent`); persisted as the `comments.intent` enum. Spec: ADR-0064 Decision 8.
- **Editing session** — an authenticated open-edit-save cycle on a report: the editor opens a version's `Report HTML schema` document, the user edits it, and saving produces a new `ReportVersion` through the existing upload/versioning pipeline (ADR-0037), re-entering the scan gate (ADR-012) exactly like any other upload — there is no mutate-in-place path. Spec: ADR-0062/0063.
- **Unified experience** — the single, app-styled `GET /<slug>/edit` surface (view.<domain>, ADR-0063 Decision 3's shipped in-viewer client) combining a View⇄Edit toggle, Comments, and Compare (a version diff) around one `Editing session`, so a `canWrite` user never has to leave the viewer to comment, review history, or compare versions. The document stays the dominant element — a single, tabbed Comments|Versions panel is hidden until the user opens it. Any untrusted, stored-report-derived HTML this surface renders outside `ReportEditor`'s own sandboxed iframe (View mode's live-doc snapshot; a Compare diff) renders through the SAME sandboxed-`srcDoc` mechanism, `buildReadOnlyIframeDocument` (`arp-editor`) — never `dangerouslySetInnerHTML`, never on the app origin. Spec: ADR-0063 "The unified experience".

## Shared kernel

Terms used identically across all four contexts.

- **UserId** — branded type for **our** `User` identifier: the `users.id` UUIDv7 PK. Exposed on the wire (when exposed) as a `user_…` **External Id** (ADR-0052). This is NOT the Clerk id — that is a separate field/type, `ClerkUserId`.
- **OrgId** — branded type for **our** `Org` identifier: the `orgs.id` UUIDv7 PK. Exposed as an `org_…` External Id (ADR-0052). NOT the Clerk id (`ClerkOrgId`).
- **ClerkUserId / ClerkOrgId** — branded types for the **third-party Clerk** identifiers, persisted in the separate `users.clerk_user_id` / `orgs.clerk_org_id` columns (unique). Used only to map a Clerk session / OAuth token to our mirrored `User` / `Org` (`findByClerk`, `findPersonalOrg`); **never serialized** on the API/MCP wire (ADR-0052). Any future vendor id follows the same `<provider><Entity>Id` field-and-type pattern.
- **External Id** — the public, prefixed form an internal entity id takes on the API/MCP wire: `<prefix>_<base62(uuidv7)>` (e.g. `folder_3QmK9v…`, `report_7Kp…`). A *reversible codec over the uuid PK* (no separate column); the branded value object's `make*Id` / `toWire` decode/encode it (ADR-0052). Self-describing + mistype-proof. Distinct from a `Slug` (a capability / possession-token, not an addressing id).
- **Resource (wire)** — the API/MCP shape of an entity (ADR-0053): a flat `snake_case` object with `object` (the type discriminator, e.g. `"report"`/`"folder"`), `mode`, its `External Id`, and its fields. No envelope nesting (not JSON:API). Errors are the RFC-9457 `Problem` instead (ADR-0040).
- **List envelope** — the API/MCP shape of a collection (ADR-0053): `{ object: "list", data: [<resource>…], has_more }`. No `total`. Returned by every list endpoint (reports, folders).
- **Cursor pagination** — keyset pagination over an entity's `External Id` (UUIDv7), newest-created first (ADR-0053): `limit` (1..100, default 20) + `starting_after` / `ending_before` (a prefixed id); `has_more` signals another page. Replaces offset/`page`. `CursorPage<T>` = `{ items, hasMore }` is the internal (pre-wire) form.
- **mode** — enum on every wire `Resource` (ADR-0053): `"prod"` on the live deployment, `"dev"` on preview/dev (from `API_KEY_ENV`). An enum (not a boolean) so it reads self-evidently on the wire and leaves room for more deployment kinds.
- **Request-Id** — a `req_…` correlation id returned in the `Request-Id` response header on every API response. As of ADR-0055 it **is** the request's OTel `trace_id`, `base62`-encoded (same `req_…` wire shape as ADR-0053, now reversible): decode a `Request-Id` → `trace_id` → the full trace + correlated logs in Grafana. Falls back to a random `req_` id when no trace is active.
- **Trace / span / span link** — observability primitives (ADR-0055, OpenTelemetry): a **trace** is one end-to-end operation (a `trace_id`), made of **spans** (timed units of work); a **span link** causally connects spans in *different* traces — used to tie a report's upload trace to its later async-scan trace across the pg-boss pipeline. Telemetry is I/O, so it lives only in adapters + apps (ADR-0024), never in `domain`/`application`.
- **Timestamp** — UTC `Date` (millisecond precision). All persisted timestamps are stored as Postgres `timestamptz`.

## Cross-cutting infrastructure

Terms for infra that sits alongside the four bounded contexts rather than inside one of them (mirrors `docs/db-design.md`'s "`outbox` / `audit_log` are cross-cutting infrastructure").

- **Audit log** — the append-only `audit_log` table: a who-did-what-to-what record for every **user-initiated, org-scoped** mutation. Written **synchronously, inside the same transaction as the mutation** (commit-last atomicity, ADR-0037 §5) by the `AuditLogger` port (ADR-0070) — NOT via async event fan-out off the domain-event stream. System- or webhook-driven state transitions (scan verdicts, user-deletion, identity provisioning) are deliberately excluded and stay on the event stream (`docs/events.md`) instead.
- **Audit action** — the closed `AuditAction` vocabulary (`packages/application/src/audit.ts`) an `Audit log` row's `action` column is written from: `report.*`, `folder.*`, `acl.set`, `grant.write.*`, `comment.*`, `api_key.*` — one member per audited user-initiated mutation use case (ADR-0070). `target_type`/`target_id`/`action` are free `text` at the DB layer; the closed union is an application-layer discipline, not a DB enum.

## Domain events

Event names are the contract; their full catalog (emitter, subscribers, payload) lives in `docs/events.md`. Events are facts in past tense.

- Reports & Folders emits: `ReportVersionUploaded`, `ReportPublished`, `AclChanged`, `CollaboratorGranted` (retired, ADR-0060 — write grants shipped deliberately without a replacement event).
- Identity & Access emits: `UserCreated`, `ApiKeyUsed`, `ApiKeyAnomalyDetected`.
- Abuse & Moderation emits: `ReportVersionScanned`, `AbuseReported`, `ReportTakenDown`, `CspViolationReported`.
- Authoring & Collaboration emits: `CommentAdded`, `CommentResolved`.
